元数据（metadata lock）锁培训记录

mysql中锁的分类

元数据锁介绍

MDL（metadata lock），即元数据锁，从MySQL 5.5开始引入，他是基于表元数据（表结构）的锁， MDL锁是为了保证并发环境下元数据和表数据结构的一致性。
如果有事务对表加了MDL锁，那么其他事务就不能对表结构进行变更，同样对于正在进行表结构变 更的事务也不允许其他事务对表数据进行增删改查，MDL锁主要用于保证DDL操作与DML操作之间的一致性。

MDL锁用于保护表的元数据信息，在表上有活动事务（显式或隐式）的时候，不可以对元数据进行写入操作。

元数据锁是MySQL Server层的锁，表级锁，每执行一条DQL、DML、DDL语句时都会申请MDL锁，
DQL和DML操作需要MDL共享锁，DDL操作需要MDL排它锁，申请MDL锁的操作会形成一个队列，队列中写锁获取优先级高于读锁。
一旦出现写锁等待，不但当前操作会被阻塞，同时还会阻塞后续该表的所有操作。事务一旦申请到 MDL锁后，直到事务执行完才会将锁释放。MDL不仅适用于表，还适用于模式（schema）和触发器。

元数据锁的分类

重点掌握前三种,其他了解即可

序号 锁类型 说明
1 SHARED_READ 共享读锁用于读取数据，例如事务中的查询行数据
2 SHARED_WRITE 共享写锁用于修改数据，例如插入更新或删除事务中的行
3 EXCLUSIVE 排他锁可以修改字典和数据，比如alter table
4 SHARED_READ_ONLY 只读锁，常见于lock table x read
5 SHARED_NO_WRITE 共享非写锁允许读取数据并阻止其他 事务 修改数据。 用于alter table的第一阶段
6 SHARED_NO_READ_WRITE 用于访问字典，读写数据，阻塞其他事务读写数据，如lock table write
7 SHARED 共享锁用于访问字典对象而不访问数据，如create table a like b
8 INTENTION_EXCLUSIVE 意向排他锁只用于范围锁，如锁lock table write
9 SHARED_UPGRADABLE SHARED_UPGRADABLE 是一个可升级的共享元数据锁，加锁期间，允许并发查询和更新，常用在 DDL 操作的第一阶段
10 SHARED_HIGH_PRIO 仅访问字典对象，例如 desc table_a，仅对MyISAM存储引擎有效

各个场景下的冲突

MDL的读锁（共享锁）和写锁（排它锁）的阻塞关系如下：

  • 共享锁和排它锁之间相互阻塞，即同一个表上的DML和DDL之间互相阻塞。
  • 排它锁和排它锁之间互相阻塞，即两个session不能对表同时做表定义变更，需要串行操作。
  • 共享锁和共享锁之间不会因MDL产生阻塞。

可能引发的问题

(1) 即使普通的SELECT查询操作，都是要持有锁，类型是SHARED_READ。
(2) 如果当前有未提交的活动事务，不允许执行alter table这种DDL。
(3) 如果当前DDL是hang（等待无响应），其他会话即使仅做查询SELECT，都会hang。

假设如果当前有未提交的活动事务,此时如果需要执行DDL语句,会因没有获取到MDL锁而陷入等待,
同时阻塞队列中其他对表的增删改查操作,直至未提交事务提交,DDL语句执行完毕且提交事务,才会继续执行后续操作
因此极易引起系统响应慢甚至超时的情况。

规避方法

程序开发侧 :

  • 规范使用事务，及时提交事务，避免使用大事务；
  • 少用工具开启事务进行查询，图形化工具要及时关闭；
  • DDL操作放在业务低峰期执行，通过SQL审核系统定时晚上执行；
  • 若以查询为主，建议默认不开启事务；

运维侧 :

  • 监控元数据锁

元数据锁故障处理方法 :

  • 找到阻塞元数据锁的源头，将事务提交或回滚，或等待事务执行完成；
  • 终止申请元数据锁的语句，如终止alter table语句；
  • 重启应用服务释放元数据锁；
  • 重启数据库实例释放锁。
